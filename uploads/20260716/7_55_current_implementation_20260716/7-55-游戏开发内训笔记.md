# 《7:55》游戏开发内训笔记

日期：2026-07-11  
适用对象：前端开发、游戏逻辑开发、关卡策划、测试与内容制作  
项目形态：手机伪系统解谜 + 竖屏街机小游戏 + 横屏校园 RPG  
当前版本：第一关已实现，第二章和第三章 V1.0 已实现

## 1. 项目概述

《7:55》以浙大校园生活为叙事环境。玩家会在手机桌面、CC98、浙大钉、图书馆、浙大体艺和校园地图之间切换，通过信息检索、物品组合、拖放、路线验证、碰撞避让等操作推进章节。

项目的核心工程目标有三个：

1. 多个应用页面共享同一份可持久化游戏进度。
2. React 界面、Phaser 动作场景、音频和演出层能够独立工作。
3. 最终产物可通过单个 `demo/index.html` 离线打开，适合演示和交付。

## 3. 整体架构

### 3.1 双运行时

项目有两种顶层运行模式：

- `phone`：React 在固定 `430×930` 逻辑视口内渲染手机页面。
- `rpg`：Phaser 使用 `960×540` 逻辑画布渲染校园 RPG。

`App.tsx` 根据 `runtimeMode` 切换运行时。RPG 通过动态 `import()` 延迟载入，手机页面通过场景注册表切换。两种运行时共用 `GameState`、`SceneRouter` 和 `EventBus`。

### 3.2 分层责任

```text
玩家输入
  ↓
React 场景 / Phaser Scene
  ↓ 调用
Controller / puzzleRules
  ↓ 校验并写入
Zustand GameState + SaveStore
  ↓ 产生状态变化或领域事件
PresentationDirector
  ↓ 转换为稳定 cue
PresentationLayer / AudioDirector
```

其中，组件负责显示和收集输入，Controller 负责规则与状态转移，Director 负责演出。页面动画完成、音频播放完成都不直接改写谜题进度。

### 3.3 状态与存档

`GameState` 管理的内容包括：

- 当前运行时和场景。
- 网络、主题、亮度等手机系统状态。
- 数字碎片、道具和组合物品。
- 第一关启动流程。
- 第二章谜题阶段和阅读证据。
- 第三章的解锁、尝试次数和最佳成绩。

`SaveStore` 对 `localStorage` 中的 JSON 进行逐字段验证。枚举、数字范围、布尔值和数组都有归一化逻辑。旧存档、缺字段存档或异常 JSON 会退回到安全默认值。

## 4. 三类核心玩法的实现

### 4.1 手机伪系统解谜

手机系统使用共享 `PhoneShell`、`StatusBar`、道具栏和提示层。每个 App 页面只填充内部可用区域，无权改动外层手机尺寸。

第一关主要用于建立这套基础交互：

- 闹钟唤醒与桌面入口。
- 微信、校园卡、控制中心、天气等页面中的线索收集。
- 道具组合、数字收集、网络和屏幕方向状态变化。
- 手机与校园 RPG 之间的路由切换。

实现要点是将关键进度放入共享状态。动画是可重播的界面反馈，不承担关卡通关判定。

### 4.2 跨 App 证据链解谜

第二章“期末周：三分钟离座”的主线跨越图书馆、CC98、浙大体艺和照片证据层。主要操作包括：

1. 预约图书馆 `022` 座位，触发三分钟离座后的座位丢失。
2. 进入 63 层虚构帖，折叠 18 条纯 `ac01` 回复。
3. 核对 5 个分散楼层的对话，进入两级旧引用，拼出路线顺序。
4. 在浙大体艺中拖动定位点，通过检查点并避让自行车。
5. 将轨迹证据附加到剧情帖，根据来源闭合性选择有效 `bd`。
6. 从楼主三次编辑中推导口令，设置审核参数，将 PASS 拖回 `022`。

这类谜题的主要特征是“阅读行为也属于进度”。`investigatedStoryFloors`、`citationDepth`、`routeVisited`、`appliedBdReplyIds` 等字段会持久化。玩家切换页面后，控制器仍能判断他是否真正完成了证据链。

### 4.3 Phaser 动作关

第三章“求是潮 755”在手机竖屏内运行 Phaser Canvas。玩家在三条车道之间换道，避开自行车、路障和人群，完成 `755m` 目标。

主要规则：

- 内部画布为 `390×650`，使用 `Phaser.CANVAS` 和 Arcade Physics。
- 三条车道中心为 `88 / 195 / 302`。
- 支持 A/D、左右方向键、底部按钮和点击赛道左右半区。
- 换道 tween 时长为 `105ms`。
- 碰撞扣除一次机会，并进入 `900ms` 无敌时间。
- 障碍速度和生成密度随距离递增。
- 同一波最多占用两条车道，生成器会拒绝无解组合。
- 窗口失焦或页面隐藏时暂停，恢复后丢弃第一个异常时间帧。
- 销毁组件时清理 Phaser 实例、输入监听、计时器和运行时快照。

## 5. 主要开发难点

### 难点一：跨页面、跨引擎的状态一致性

**问题**

一个章节可能经过多个 React 页面和 Phaser 场景。如果每个页面自行保存进度，页面切换、刷新或运行时切换后容易丢失状态，也可能出现跳步。

**解决方案**

- 使用唯一 `GameState` 作为进度权威。
- React 通过 `useSyncExternalStore` 订阅，Phaser 通过 bridge 读写。
- Controller 验证证据门槛和终止条件。
- `SaveStore` 对读取的本地数据进行白名单归一化。

**可复用经验**

当一个功能需要跨路由、跨模态或跨引擎继续时，要先定义状态所有权，再开始写页面。

### 难点二：谜题既要防跳步，又要保持可理解

**问题**

第二章含有楼层阅读、引用追踪、路线拼接、证据附加和参数审核。只检查最终答案会让玩家通过试错跳过调查过程；强制玩家点击过多控件又会造成流程负担。

**解决方案**

- 将主推理拆成多个证据载体，分散到不同楼层、用户和 App。
- 关键阅读行为写入共享存档，临时展开动画留在组件内。
- 诱饵选项给出可解释的失败原因，如时间戳矛盾、附件缺失和来源未闭合。
- 失败提示逐步提供“错误类型 → 结构规则 → 证据类型”，不直接输出答案。

**可复用经验**

关卡的验证对象应包括“玩家获得结论的必要过程”。过程门槛要少且可解释。

### 难点三：React 与 Phaser 的生命周期衔接

**问题**

Phaser 使用持续帧循环，React 使用组件挂载和卸载。反复进入小游戏、重试、切换标签页或窗口失焦时，可能出现重复实例、重复键盘监听、计时器泄漏和超大 `deltaMs`。

**解决方案**

- 将 Phaser 实例创建与 React 组件挂载绑定。
- 卸载或重试时统一销毁实例、监听器、计时器和快照。
- `BikeArcadeLifecycle` 同时记录 `document-hidden` 和 `window-blur` 两种暂停原因。
- 恢复后丢弃首帧，常规帧的时间增量上限为 `100ms`。

**可复用经验**

任何嵌入 React 的长运行时实例都需要明确的创建、暂停、恢复和销毁协议。

### 难点四：音频、字幕、动画与游戏逻辑解耦

**问题**

音频存在自动播放限制、资源缺失、加载失败和时长偏差。如果通关条件等待音频或动画回调，玩家可能无法继续。

**解决方案**

- 游戏逻辑只发布领域事件和状态变化。
- `PresentationDirector` 将多个来源归一化为稳定 `presentation_cue`。
- `PresentationLayer` 和 `AudioDirector` 独立消费 cue。
- 音频时间线写在 JSON 中，支持偏移、音量、循环、字幕、背景音乐压低和单次播放。
- 关闭场景时取消已排队 cue，停止旁白和章节音乐。
- 播放失败被局部处理，游戏状态仍然继续。

**可复用经验**

演出系统可以订阅游戏逻辑，不应成为游戏逻辑的前置条件。

### 难点五：固定逻辑视口下的多页面适配

**问题**

参考图的比例并不统一，项目又需要所有手机场景保持 `430×930`。如果每个页面自行缩放外框，切换页面时会发生尺寸跳动，点击区也容易与控件脱离。

**解决方案**

- `PhoneShell` 独占手机宽高、响应式缩放、边框和外层居中责任。
- 页面只使用 `width: 100%` 和 `height: 100%` 填充内部。
- 顶部 `40px` 始终保留给共享状态栏。
- 参考图只用于视觉核对，交互页面使用 React/CSS 原生重建。
- 点击区绑定到真实渲染控件，不依赖视口猜测坐标。

**可复用经验**

嵌套式应用需要先确定外层容器的尺寸所有权。内容适配发生在容器内部。

### 难点六：随机关卡的可玩性与可测试性

**问题**

随机障碍可能生成无法躲避的封路组合。同时，完全依赖画面和真实时间会让自动化测试难以重现问题。

**解决方案**

- 随机生成器先应用“每波最多两车道”和“保留换道时间窗口”约束。
- 开场保护期和同屏障碍上限用于控制局部压力。
- 距离、里程碑、无敌和障碍规则使用独立纯函数测试。
- 运行时向外暴露深拷贝快照，用于检查阶段、距离、生命、车道和障碍。

**可复用经验**

随机系统需要先定义不可违反的规则，再在可行范围内随机取样。

### 难点七：离线单文件交付

**问题**

开发环境中的模块路径、懒加载、音频、图像和 JSON 资源在 HTTP 服务下可以正常运行，直接打开 HTML 时仍可能失效。

**解决方案**

- 通过 `vite-plugin-singlefile` 生成 `demo/index.html`。
- 使用 Vite `import.meta.glob` 集中解析音频资源。
- 构建前运行 TypeScript 类型检查。
- 交付检查包括 `typecheck`、全量测试、单文件构建和真实浏览器完整导航。

**可复用经验**

交付形式会反向约束资源引用和模块分割方式，需要在项目早期纳入架构设计。

## 6. 测试与质量保证

项目当前有 35 个 `*.test.*` 文件，测试分为四层：

1. **规则层**：路线顺序、审核参数、车道限制、距离和里程碑。
2. **控制器层**：章节阶段转移、物品收集、网络切换和签到流程。
3. **组件层**：CC98 过滤、引用链、`bd` 选择、路线审核和照片证据。
4. **集成层**：`App`、场景路由、音频 cue、RPG bridge 和地图模型。

常用验证命令：

```bash
npm run typecheck
npm run test:run
npm run build:single
```

代码检查不能覆盖所有交互问题。输入与视觉修改还需要在真实浏览器中检查：

- `430×930` 逻辑视口和一个缩放移动端视口。
- 页面切换时手机外框是否稳定。
- 拖放、触摸、键盘和点击热区是否对齐。
- Phaser Canvas 是否持续非空白渲染。
- 离线 `demo/index.html` 是否能完成导航和关键交互。

## 7. 团队协作时的代码边界

- 手机场景放在 `src/scenes/phone/`。
- RPG 场景放在 `src/scenes/rpg/`。
- 跨场景状态、路由和事件放在 `src/core/`。
- 共享玩法逻辑和章节 Controller 放在 `src/modules/`。
- 对话、谜题、时间线和地图数据放在 `src/data/`。
- 共享手机外壳控件放在 `src/components/`。
- 场景可以依赖 core、modules、components 和 data，不跨目录引用其他场景的内部文件。
- 任何新机制在写入“已完成”前，必须至少执行一次真实验证。

## 8. 新成员建议上手顺序

1. 阅读 `CLAUDE.md` 和 `docs/framework-spec.md`，理解视口、场景和状态所有权。
2. 从 `src/App.tsx` 查看双运行时入口。
3. 阅读 `src/core/GameState.ts`、`SaveStore.ts` 和 `SceneRouter.ts`。
4. 选择一条章节链，按“页面 → Controller → State → Event → Presentation”追踪一次。
5. 阅读对应测试，确认哪些条件属于强制规则。
6. 运行 `npm run typecheck` 和 `npm run test:run`。
7. 修改界面或行为后，构建单文件版并在真实浏览器中完成一次关键链路。

## 9. 内训学习与实践收获

### 9.1 知识树：先建立学习顺序

在实际开发中，可以将本项目的知识树分为四层：

```text
基础层
├─ TypeScript 类型、React 组件、CSS 布局
└─ Git、npm、Vite 构建

工程层
├─ 目录边界、共享状态、路由
├─ 数据驱动、本地存档、异常恢复
└─ 单元测试、组件测试、构建验证

玩法层
├─ 状态机、证据门槛、反馈分级
├─ Pointer/Keyboard Input、拖放、碰撞
└─ Phaser Scene、Arcade Physics、关卡难度曲线

系统层
├─ React/Phaser 双运行时
├─ 领域事件、演出 cue、音频调度
└─ 离线单文件交付和真实浏览器验收
```

这棵树解决了“先学什么”的问题。如果基础层尚不稳定，直接进入跨引擎状态同步，调试成本会明显上升。

### 9.2 知识图谱：理解概念之间的关系

本项目中最有价值的图谱主链是：

```text
玩家行为
→ 场景组件
→ Controller 规则验证
→ GameState 状态转移
→ SaveStore 持久化
→ EventBus 事件
→ PresentationDirector 演出 cue
→ 视觉和音频反馈
```

知识树用于安排先后顺序，知识图谱用于表达依赖和数据流。开发一个新关卡时，可以先在知识树上判断团队是否具备所需能力，再在图谱上检查新机制会影响哪些现有节点。

### 9.3 教学项目实践收获

1. **先写状态合同，再写页面**：定义进度字段、允许的阶段和终止条件后，多人开发时的冲突更少。
2. **教学项目要有可观测输出**：状态、事件历史、运行时快照和测试结果都可用于说明机制是否正常。
3. **项目验收反向推动学习**：单文件交付、刷新后保留进度、手机触摸可用等指标，会迫使开发者理解资源、状态和生命周期。
4. **小步验证比集中调试更有效**：规则函数、Controller、组件和完整玩法分层验证，容易定位问题所在层级。

### 9.4 当前疑问

- 知识图谱是只用于内训导航，还是要与代码模块、测试和验收记录双向链接？
- 学习进度的最小粒度应当是“完成一节内容”，还是“独立完成一次可验证实践”？
- 教学项目的自动化测试覆盖率和真实浏览器验收应如何分配比重？
- 对于具有随机性的游戏关卡，验收是否需要固定随机种子和最长可重现路径？

### 9.5 对内训的建议

1. 每个知识节点同时给出“先修节点、最小实践、验收证据、常见失败”四个字段。
2. 使用同一个教学项目连续承载基础与进阶任务，减少重复搭建项目的时间。
3. 每个阶段设置可执行验收脚本和一条人工操作链，保留完整的失败信息。
4. 评分时区分“会使用 API”、“能解释运行机制”和“能处理边界条件”三个层级。
5. 要求学员保留问题闭环记录，避免项目最终只有成品截图和结果描述。

## 10. 踩坑、提问与最终解决方案

| 踩坑或问题 | 开发中提出的问题 | 根因 | 最终解决方案 | 验证方式 |
| --- | --- | --- | --- | --- |
| 页面切换后谜题进度可能丢失 | 哪些状态可以留在组件内？ | UI 状态和游戏进度混合 | 动画、临时选中留在本地；证据、门槛和章节阶段写入 `GameState` | Controller 测试 + 切换页面回访 |
| 存档结构升级后旧数据导致异常 | 能否直接将 `localStorage` JSON 当作 `GameState`？ | 外部数据未验证 | `SaveStore` 对枚举、数值范围、数组和嵌套对象逐层归一化 | 存档单测 + 异常 JSON 回退 |
| 玩家可能通过猜答案跳过调查 | 通关判定只检查最终值是否足够？ | 缺少证据过程记录 | 保存已阅楼层、引用深度、路线记录和有效回复 ID，Controller 统一审核 | 谜题规则测试 + 故意跳步测试 |
| 过滤结束后自动跳到答案位置 | 便捷导航是否削弱了阅读任务？ | 把任务提示实现成了答案导航 | 保持原滚动位置，让玩家根据回复关系主动定位 | CC98 组件测试 + 真实阅读链 |
| 音频自动播放失败时流程停滞 | 音频结束回调能否作为状态转移信号？ | 演出被设计成游戏逻辑的依赖 | 状态先转移，演出独立消费 cue，播放异常局部处理 | `AudioDirector` 与 `PresentationDirector` 测试 |
| 重进 Phaser 页面后可能出现重复监听 | React 卸载时需要清理哪些 Phaser 资源？ | 两套生命周期没有统一退出协议 | 销毁 game、input listener、timer 和 snapshot，重试时创建新 `runId` | 反复进入/退出测试 + runtime 测试 |
| 切回页面后障碍瞬间跨越画布 | 後台页面的时间是否应计入游戏帧？ | 恢复首帧 `deltaMs` 过大 | 失焦/隐藏时暂停，恢复首帧归零，后续帧上限 `100ms` | `BikeArcadeLifecycle` 单测 |
| 随机障碍封死三条车道 | 如何保证随机关卡始终存在可行操作？ | 先随机生成，缺少约束校验 | 每波最多两车道，并根据 `105ms` 换道时间拒绝无解组合 | 生成规则测试 + 实际试玩 |
| 参考图与手机视口比例不一致 | 应缩放手机外框还是重排内容？ | 外层尺寸所有权不清晰 | 固定 `430×930`，使用 React/CSS 重建内部布局，热区跟随控件 | 两种视口视觉验收 |
| 开发服务可运行，直接打开交付 HTML 可能失效 | 本地开发通过能否代表交付通过？ | 交付环境与开发环境的资源协议不同 | 单文件构建，并对 `demo/index.html` 执行真实浏览器验收 | `build:single` + 完整导航链 |

## 11. 对内训内容的延伸学习

### 11.1 状态机与领域建模

当前项目使用 TypeScript 枚举字段和 Controller 表达状态转移。后续可以学习显式状态机、状态转移图、守卫条件和基于模型的测试，并比较引入状态机库对可读性和维护成本的影响。

### 11.2 知识图谱与代码资产关联

可将“技术概念、代码模块、关卡机制、测试用例、验收证据”建模为节点，将“依赖、验证、消费、产生、阻塞”建模为关系。延伸实践应先选择一条章节链做小规模图谱，验证导航和故障定位价值后再扩大范围。

### 11.3 程序化关卡与可解性

第三章已经使用受约束的随机障碍。后续可学习固定随机种子、可达性检查、难度评分、玩家输入回放和属性测试，将“本局可通过”变成可重复检查的条件。

### 11.4 游戏可访问性

项目已考虑指针热区、多种输入方式和减少动效偏好。后续可继续学习键盘全流程、色彩对比度、焦点管理、非颜色反馈、字幕时间和简化操作模式。

### 11.5 观测性与回放调试

当前已有事件历史和 Phaser 快照。进阶方向可扩展为带时间戳的输入记录、状态差异、随机种子和关键 cue 日志，用于重现“某次试玩为什么失败”。

## 12. 思考题

6. 如果项目未来加入联机或服务端存档，当前 Zustand + `localStorage` 架构中哪些边界需要重新设计？

8. 单文件交付和代码分割之间存在哪些冲突？在什么场景下应放弃单文件交付？

## 15. 阶段性结论

当前实现已经验证了三项能力：

1. React 手机界面和 Phaser 场景可以共享游戏进度。

3. 音频和演出失败时，核心状态机仍能继续。

上述结论均有当前代码和测试用例支持，适用范围限定在《7:55》当前已实现章节。若需推广到更大规模、联机或服务端存档项目，还需要新的独立验证。
