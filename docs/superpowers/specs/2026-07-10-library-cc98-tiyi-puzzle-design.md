# 图书馆 × CC98 × 浙大体艺跨应用谜题设计

日期：2026-07-10  
确认方案：C，最多两轮跨应用往返  
目标时长：8–12 分钟

## 目标

将现有“座位丢失 → 查看提示 → 恢复座位”的按钮流程升级为可操作的解谜关卡。玩家需要在图书馆、CC98、浙大体艺和照片证据之间传递信息，完成过滤、引用追踪、轨迹拖拽、顶帖、审核修正和道具投放。

## 核心约束

- 全局逻辑尺寸保持 `430×930`，场景不得修改手机外框。
- 保留现有 React、TypeScript、Zustand、EventBus 和 `AudioDirector`。
- 谜题状态决定进度；音频播放成功与否不能改变状态。
- 失败只重置当前小机关，不回退已获得线索。
- 跨应用往返最多两轮，手机主页顶部持续显示线索数量。
- 所有论坛文本与用户名保持虚构。
- 照片证据只在本地游戏中生成和使用，不向外部网站提交。
- 支持 `prefers-reduced-motion`，关闭复杂位移后仍可完成谜题。

## 完整流程

### 1. 图书馆：座位丢失

玩家预约 `022` 后点击“离开 3 分钟”。座位蓝色高亮先熄灭，再以红色闪烁。预约按钮分裂成两行文字：“原座位已被预约”和“去 CC98 查临时离座规则”。

输出：`seat_lost`。

### 2. CC98：过滤纯 ac01

剧情帖初始显示 63 个楼层，其中 18 层为纯 `ac01`。玩家开启“隐藏纯 ac01”，楼层卡片以 45ms 间隔向上折叠，隐藏计数停在 18，第 47 楼进入可见区域。

错误反馈：关闭过滤器只恢复楼层，不清空进度。  
输出：`floor_47_found`。

### 3. CC98：追踪引用链

第 47 楼引用第 12 楼，第 12 楼引用主楼。玩家依次点击引用框，页面使用短距离水平滑动展示上一层。错误点击普通楼层只震动引用边框。

输出路线顺序：`北门 → 桥 → 二层南`。

### 4. 浙大体艺：拖动绿色定位点

现有体艺背景上覆盖路线谜题。玩家拖动绿色定位点依次经过三个检查点。自行车队列横向移动，只影响本次拖拽；碰撞后定位点回到最近检查点。

成功后轨迹线变蓝，系统生成一张本地截图证据。截图自动进入照片弹层，并作为 CC98 本地剧情帖附件。

输出：`route_evidence_ready`。

### 5. CC98：bd 顶帖

附带轨迹截图的帖子位于第 4 名。玩家在三个有效回复上执行 `bd`，每次使帖子上移一位。纯表情回复和重复回复不会增加热度。

达到第 1 名时，排名牌从 `04` 逐格翻到 `01`，顶部出现“十大后续”。

输出：`top_ten_reached`。

### 6. CC98：只看楼主与恢复口令

进入后续帖后，玩家开启“只看楼主”，18 条水贴折叠。楼主的三次编辑记录分别保留数字 `7`、`47`、`3`，组成恢复口令 `7-47-3`。

输出：`recovery_code_found`。

### 7. 浙大体艺：审核回退

轨迹先显示“完成”，800ms 后回退为“待审核”。玩家使用恢复口令调整三个参数：配速 `7`、打卡次数 `47`、范围等级 `3`。每个参数使用现有像素按钮或步进器，不输入自由文本。

输出：`route_audit_passed` 和“临时通行记录”证据。

### 8. 图书馆：恢复座位

玩家回到原选座页，将“临时通行记录”从线索条拖到 `022` 座位。命中后红色闪烁停止，蓝色高亮恢复，预约状态变为已确认。

输出：`seat_recovered`，章节结束。

## 状态模型

`LibraryFinalsPhase` 扩展为：

```ts
type LibraryFinalsPhase =
  | "idle"
  | "seat_reserved"
  | "seat_lost"
  | "cc98_filtering"
  | "floor_47_found"
  | "route_order_found"
  | "route_active"
  | "route_evidence_ready"
  | "top_ten_rising"
  | "top_ten_reached"
  | "recovery_code_found"
  | "route_audit_pending"
  | "route_audit_passed"
  | "seat_recovered";
```

附加状态集中在 `LibraryFinalsPuzzleState`：

```ts
interface LibraryFinalsPuzzleState {
  ac01FilterEnabled: boolean;
  hiddenAc01Count: number;
  citationDepth: 0 | 1 | 2;
  routeVisited: Array<"north_gate" | "bridge" | "south_second_floor">;
  routeEvidenceReady: boolean;
  bdCount: 0 | 1 | 2 | 3;
  recoveryCode: string | null;
  auditPace: number;
  auditCount: number;
  auditRadius: number;
  clueIds: string[];
}
```

## 组件边界

- `LibraryFinalsController`：唯一状态推进入口，校验前置条件并发领域事件。
- `Ac01FilterPuzzle`：显示与过滤楼层，不直接修改全局状态。
- `CitationChainPuzzle`：处理引用深度和路线顺序。
- `TiyiRoutePuzzle`：处理 Pointer Events、检查点和自行车碰撞。
- `TopTenRisePuzzle`：处理有效 `bd`、排名变化和后续帖解锁。
- `RouteAuditPanel`：三个离散参数的审核修正。
- `PhotoEvidenceOverlay`：显示自动生成的轨迹截图，不承担独立剧情。
- `QuestClueStrip`：显示当前线索并提供最终拖放源。
- `AudioDirector`：只订阅事件并播放配置中的声音。

## 动画与演出时间轴

### 座位丢失，0–1600ms

- `0ms`：按钮按下，确认音。
- `120ms`：蓝色座位高亮熄灭。
- `260ms`：座位红色闪烁两次，边框错位 2px。
- `520ms`：播放拒绝音，页面轻震 240ms。
- `760ms`：旁白进入，音乐降低到 30%。
- `1450ms`：CC98 图标在主页产生两次像素脉冲。

### ac01 过滤，0–1100ms

- `0ms`：开关发出机械拨动音。
- `100–910ms`：18 个楼层以 45ms 间隔折叠。
- 每折叠 6 层播放一次轻量计数音，共 3 次。
- `960ms`：第 47 楼边框由灰变蓝。
- `1040ms`：播放线索确认音，不立即播放旁白。

### 引用链，每次 420ms

- 引用框向左滑出 18px，新楼层从右进入。
- 深度从 0 到 2 时，音高分别为 1.0、1.08、1.16。
- 路线顺序完成后，三个地点标签依次落入线索条。

### 体艺轨迹

- 定位点跟随 Pointer Events，不使用缓动，保证手感直接。
- 轨迹线每 8px 采样一次，视觉使用 4px 像素折线。
- 自行车队列以 1800–2600ms 不同周期横向移动。
- 碰撞时冻结 100ms、播放短刹车音，再回到最近检查点。
- 第三个检查点命中后 700ms 内完成轨迹染蓝和截图闪白。

### bd 顶帖，每次 560ms

- `0ms`：回复按钮下压。
- `80ms`：帖子卡片上移一格。
- `260ms`：排名牌翻面。
- `420ms`：浏览与回复数字滚动一次。
- 第三次成功后追加 900ms 的“01”印章演出。

### 审核回退，0–2100ms

- `0ms`：绿色完成状态出现。
- `800ms`：提示音反向播放，绿色条倒退。
- `1050ms`：状态变成黄色“待审核”。
- 参数正确后逐项亮起，最后播放四音确认句。

### 座位恢复，0–1800ms

- 拖放命中后证据图标缩入座位中心。
- 红色闪烁停止，蓝色从中心向外扩展。
- `600ms`：预约按钮恢复。
- `820ms`：播放恢复音和最后一句旁白。
- `1600ms`：章节完成标记出现。

## 音频设计

### 配乐

六个阶段使用明显不同的配乐：

1. `music_library_lost`：低频、稀疏提示音。
2. `music_cc98_investigation`：短促点击和轻快脉冲。
3. `music_tiyi_route`：稳定节拍，方便拖拽操作。
4. `music_cc98_top_ten`：速度提高，加入通知音色。
5. `music_tiyi_audit`：节拍减少，突出参数反馈。
6. `music_library_recovered`：明亮短循环，章节结束后淡出。

应用切换使用 300ms 交叉淡化。旁白播放时音乐降到原音量的 25–35%。

### 旁白

同一旁白角色保持音色一致，共 9 条：座位丢失、过滤完成、引用完成、轨迹开始、证据生成、十大达成、口令发现、审核通过、座位恢复。旁白只在状态变化后播放，不覆盖拖拽和参数操作。

### 音效

- 座位拒绝、页面轻震。
- 过滤开关、折叠计数、楼层确认。
- 引用滑动三种音高。
- 检查点命中、自行车掠过、碰撞刹车。
- 截图快门、照片落入附件。
- `bd` 上移、排名翻牌、十大印章。
- 审核反向、参数步进、审核通过。
- 证据拖放、座位恢复。

音效事件与文件名写入 JSON；缺失文件只跳过播放。

## 旁白草案

1. 座位丢失：“你离开了三分钟。系统用其中两分钟证明你从未拥有过这个座位。”
2. 过滤完成：“十八层内容已经隐藏。有效信息终于获得了屏幕使用权。”
3. 引用完成：“路线找到了。它只经过两个旧帖和一次考古。”
4. 轨迹开始：“请拖动定位点。自行车不会主动配合你的学业安排。”
5. 证据生成：“截图已保存。照片现在具备了行政用途。”
6. 十大达成：“排名第一。事件本身没有变化，字体已经更大了。”
7. 口令发现：“三个编辑记录，一条恢复口令。规则完整得很分散。”
8. 审核通过：“四十七次记录有效。系统同意承认刚才发生过。”
9. 座位恢复：“座位恢复了。请在下一次规则更新前坐下。”

## 错误与恢复

- 页面刷新后保留已完成阶段和线索。
- 路线碰撞只回最近检查点。
- 无效 `bd` 不消耗有效次数，只播放拒绝反馈。
- 审核参数错误时指出错误项数量，不直接显示答案。
- 拖放座位失败时证据回到线索条。
- 音频自动播放受限时显示字幕，游戏继续。
